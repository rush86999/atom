## Goal
Make the full app (`main_api_app.py` — the production Docker entrypoint) actually work: stop silently swallowing ~280 broken imports, recover every route that has an existing router file, build thin routers for integrations that have a service but no routes, prune truly-dead imports, and tighten the safety net so this can't recur silently. Verify with a smoke test + journey test.

## Confirmed diagnosis (data gathered)
`main_api_app.py` references ~294 `api.*` modules. Of those:
- **12 resolve correctly** already
- **97 EXACT path mismatches** — file exists at a real path (e.g. `api.routes.canvas_routes` → `api/canvas_routes.py`, `api.routes.integrations.notion_routes` → `integrations/notion_routes.py`)
- **30 NEAR matches** — file exists under a variant name (`_api.py`/`_endpoints.py`/archived), e.g. `api.routes.workflow_routes` → `core/workflow_endpoints.py`, `api.routes.feedback_enhanced_routes` → `api/feedback_enhanced.py`
- **155 DEAD** — no router file anywhere (genuinely never-built: spotify routes, zoho routes, razorpay, sandbox, scim, debug_*, test_*, etc.)

`safe_import_router()` (line 209) swallows every `ImportError` and returns an empty `APIRouter()`, so all 282 failures degrade to silent 404s. The app "boots" (148 routes) but ~250 intended endpoints are dead.

## Execution plan (staged, each stage independently verifiable)

### Stage 1 — Fix the 97 EXACT path mismatches (mechanical, safe)
For each, rewrite the import string from `api.routes.X` to the verified real module path. Done as a scripted transform over `main_api_app.py` using the exact mapping table I've compiled (97 entries). Examples:
- `safe_import_router("api.routes.canvas_routes")` → `"api.canvas_routes"`
- `safe_import_router("api.routes.integrations.notion_routes")` → `"integrations.notion_routes"`
- `safe_import_router("api.routes.admin.business_facts_routes")` → `"api.admin.business_facts_routes"`
Each replacement verified against the real file's existence.

### Stage 2 — Fix the 30 NEAR matches (per-item judgment)
For each, open the candidate file, confirm it exports a `router`, and map the import to it. Examples:
- `api.routes.workflow_routes` → `core.workflow_endpoints`
- `api.routes.feedback_enhanced_routes` → `api.feedback_enhanced`
- `api.routes.integrations.atom_sales_live_routes` → `integrations.atom_sales_live_api`
For any candidate that does NOT export `router` (e.g. it's a service, not a router), move it to Stage 4 (prune) instead.

### Stage 3 — Build thin routers for integrations with a service but no routes (~15-20)
From the DEAD integration family where a `_service.py`/adapter exists but no router: create a minimal router module that exposes the service via a couple of endpoints (status + core read action). Candidates: zoho_crm, zoho_books, zoho_inventory, zoho_mail, zoho_projects, spotify, razorpay, sagehr, segment, azure_devops. Each new router follows the existing pattern in `integrations/<name>_routes.py`. (Exact set finalized during execution based on which have a usable service class.) These get imported with corrected paths.

### Stage 4 — Prune the truly-DEAD imports (~120-140)
Remove the import lines + their `app.include_router(...)` lines for modules that have NO file and NO service to build on (debug_*, test_*, sandbox_execution, scim, sdlc, industrial, openclaw_*, marketplace variants that are archived, etc.). These are features that were never built — leaving the import just creates a silent no-op. Pruning makes `safe_import_router` honest.

### Stage 5 — Tighten `safe_import_router` (fail loud in dev)
Modify the helper so that in non-production (`ENVIRONMENT != production`), an `ImportError` for a module NOT in an explicit allowlist of "genuinely optional" integrations **raises** instead of returning an empty router. Production keeps graceful degradation (so a missing optional dep doesn't crash the pod). This prevents the original bug from recurring: a typo'd import path will fail the test suite, not silently 404.

### Stage 6 — Verify
1. `import main_api_app` → confirm it loads with no swallowed ImportError for any non-optional router; count routes (target: far above 148, ideally 350+).
2. Smoke test via TestClient: hit `/health/live`, `/`, `/openapi.json`, and a Notion endpoint (`/api/v1/integrations/notion/health`).
3. Add a journey test `tests/integration/journeys/test_full_app.py` that builds the full app and asserts (a) it boots, (b) route count is above a floor, (c) health responds. This guards against future regression.
4. Confirm the existing minimal-app journey suite still passes (128/1).

## Risks & mitigations
- **Wrong near-match mapping** → each Stage 2 fix verified by checking the file exports `router` before mapping.
- **New router with broken service** → thin routers wrap existing service methods with try/except; if a service can't init, the endpoint returns 503, not a crash.
- **Import side-effects** → `main_api_app` import order matters; staged changes verified by re-importing after each stage.
- **Scope of "build missing"** → I'll build routers only where a real service class already exists. Pure-vapor features (no service, no spec) get pruned, not invented.

## What I will NOT do
- Rewrite or "fix" the 155 dead modules that have no service/spec (that's new product development, not "make the app work").
- Touch `main.py` (the minimal app stays the dev/test entrypoint).
- Lower any test/coverage thresholds.
- Commit/push unless you ask (I'll ask at the end).
