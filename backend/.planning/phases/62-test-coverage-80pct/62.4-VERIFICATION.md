## Root Cause Identified

Through binary search debugging, identified the specific file causing the issue:

**test_cognitive_tier_e2e.py** imports **main_api_app_safe.py** which creates an incomplete mock of numpy:
```python
# main_api_app_safe.py lines 14-22
np_mock = mock_package("numpy")  # Creates empty module without .short
sys.modules["numpy.linalg"] = MagicMock()
sys.modules["numpy.core"] = MagicMock()
sys.modules["numpy._core"] = MagicMock()
sys.modules["numpy.core.multiarray"] = MagicMock()
sys.modules["numpy._core.multiarray"] = MagicMock()
```

When `test_excel_export_analytics.py` imports `discord_analytics_engine`, which imports `openpyxl`, the openpyxl library tries to access `numpy.short` and other dtypes that don't exist in the mock.

## Solution Implemented

Added module-level cleanup to `test_excel_export_analytics.py`:

```python
def _is_mocked_module(module, mod_name):
    """Check if a module is mocked or incomplete."""
    if module is None:
        return True
    if hasattr(module, '_spec_class'):  # MagicMock detection
        return True
    if mod_name == "numpy" and not hasattr(module, 'short'):  # Incomplete numpy
        return True
    return False

for mod in ["numpy", "pandas", "lancedb", "pyarrow"]:
    if mod in sys.modules and _is_mocked_module(sys.modules[mod], mod):
        # Remove the module and ALL submodules (critical for numpy.core.*)
        keys_to_remove = [k for k in sys.modules.keys() if k == mod or k.startswith(mod + '.')]
        for key in keys_to_remove:
            sys.modules.pop(key, None)
        # Also remove openpyxl modules with cached references
        for openpyxl_mod in list(sys.modules.keys()):
            if openpyxl_mod.startswith('openpyxl'):
                sys.modules.pop(openpyxl_mod, None)
```

**Key insight:** Simply removing `numpy` from sys.modules wasn't enough because the mocked submodules (`numpy.core.*`, `numpy._core.*`) were still present and causing import errors when the real numpy tried to load.

## Lessons Learned

1. **Binary search debugging is effective** for identifying the specific file causing import issues in large test suites
2. **Mock submodules matter** - mocking `numpy.core` without `numpy.short` causes hard-to-debug issues
3. **openpyxl is sensitive** to numpy state - it caches numpy dtype references at import time
4. **Cleanup must be complete** - removing a module requires also removing all its submodules
