# Phase 62, Plan 15: Integration Tests for Critical Paths

**Objective:** Write integration tests for high-impact files using real database (SQLite in-memory) and TestClient instead of mocks, focusing on critical paths that drive business value.

**Purpose:** Phase 62's unit tests used heavy mocking that didn't cover real code. Integration tests with real DB/API exercise 2-3x more code per test. This plan targets the highest-impact files for maximum coverage gain.

**Output:** 60-80 integration tests covering critical paths, coverage increases to 40-45%.

From 62-RESEARCH.md, Section "Why Phase 62 Failed":

> "Heavy Mocking Prevents Real Code Coverage:
> - Workflow engine tests mock everything
> - Tests pass but don't exercise real workflow_engine.py code
> - Result: Tests test mocks, not production code"

And from "Integration Testing Focus":

> "Integration Tests Win When:
> - Database Operations: Unit test mocks session (tests mock), Integration test uses real SQLite (tests actual SQL)
> - ROI: 3x better (catches SQL bugs)
> - API Endpoints: Unit test mocks request/response, Integration test uses TestClient (tests routing, validation, serialization)
> - ROI: 2x better (catches routing bugs)"
