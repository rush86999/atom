<objective>
**Automate type generation in CI/CD** creating a workflow that regenerates API types on backend changes, validates TypeScript compilation, detects breaking changes, and commits the generated types to ensure frontend-mobile-desktop type synchronization.

**Purpose:** Manual type regeneration is error-prone and easily forgotten. When backend routes change, the OpenAPI spec changes but frontend types drift out of sync. This automation ensures types are always synchronized, breaking changes are detected early, and developers are notified of contract violations.

**Output:**
- CI/CD workflow (.github/workflows/api-type-generation.yml) for type generation
- Breaking change detection integration with openapi-diff (from Phase 128)
- TypeScript compilation validation after type generation
- Automated commit of regenerated types
- PR comments for breaking changes and type mismatches
</objective>

<task type="auto">
  <name>Task 1: Create CI workflow for API type generation</name>
  <files>.github/workflows/api-type-generation.yml</files>
  <action>
Create GitHub Actions workflow for automated type generation:

1. Create `.github/workflows/api-type-generation.yml`:

   ```yaml
   name: API Type Generation

   on:
     push:
       branches: [main, develop]
       paths:
         - 'backend/**/*.py'
         - 'backend/core/openapi.py'
         - 'backend/api/**/*.py'
     pull_request:
       branches: [main, develop]
       paths:
         - 'backend/**/*.py'
         - 'backend/core/openapi.py'
         - 'backend/api/**/*.py'
     workflow_dispatch:

   jobs:
     generate-types:
       runs-on: ubuntu-latest
       steps:
         - uses: actions/checkout@v4

         - name: Set up Python
           uses: actions/setup-python@v5
           with:
             python-version: '3.11'

         - name: Set up Node.js
           uses: actions/setup-node@v4
           with:
             node-version: '20'

         - name: Install backend dependencies
           working-directory: ./backend
           run: |
             python -m pip install --upgrade pip
             pip install fastapi uvicorn

         - name: Generate OpenAPI spec
           working-directory: ./backend
           run: |
             python tests/scripts/generate_openapi_spec.py -o openapi.json

         - name: Install openapi-typescript
           run: npm install -g openapi-typescript

         - name: Generate TypeScript types
           run: |
             npx openapi-typescript backend/openapi.json -o frontend-nextjs/src/types/api-generated.ts

         - name: Verify TypeScript compilation
           working-directory: ./frontend-nextjs
           run: |
             npx tsc --noEmit src/types/api-generated.ts

         - name: Detect breaking changes
           run: |
             if [ -f backend/openapi-baseline.json ]; then
               npx openapi-diff backend/openapi-baseline.json backend/openapi.json || true
             fi

         - name: Commit generated types
           run: |
             git config --local user.email "action@github.com"
             git config --local user.name "GitHub Action"
             git add frontend-nextjs/src/types/api-generated.ts
             git add backend/openapi.json
             git diff --staged --quiet || git commit -m "chore: regenerate API types from OpenAPI spec"

         - name: Upload generated types as artifact
           uses: actions/upload-artifact@v4
           with:
             name: api-generated-types
             path: |
               frontend-nextjs/src/types/api-generated.ts
               backend/openapi.json
             retention-days: 30
   ```

2. The workflow triggers on backend code changes (Python files in backend/api/, backend/core/)
   Paths filter ensures workflow only runs when API routes might have changed

3. Install openapi-typescript globally for availability without frontend npm install

4. Type check generated types to catch TypeScript compilation errors early

5. Commit changes automatically (types are source code for consumers, not build artifacts)

Do NOT modify existing ci.yml (create separate workflow for type generation)

<task type="auto">
  <name>Task 3: Add breaking change detection and PR comments</name>
  <files>.github/workflows/api-type-generation.yml</files>
  <action>
Enhance the CI workflow with breaking change detection and PR comments:

1. Update api-type-generation.yml to add breaking change detection step after "Generate TypeScript types":

   ```yaml
   - name: Check for breaking changes
     id: breaking-check
     run: |
       if [ -f backend/openapi-baseline.json ]; then
         echo "Checking for breaking OpenAPI changes..."
         npx openapi-diff backend/openapi-baseline.json backend/openapi.json > diff-output.txt 2>&1 || true

         if grep -q "Breaking changes" diff-output.txt; then
           echo "has_breaking=true" >> $GITHUB_OUTPUT
           cat diff-output.txt
         else
           echo "has_breaking=false" >> $GITHUB_OUTPUT
         fi
       else
         echo "No baseline found, creating one..."
         cp backend/openapi.json backend/openapi-baseline.json
         echo "has_breaking=false" >> $GITHUB_OUTPUT
       fi

   - name: Comment breaking changes on PR
     if: failure() && github.event_name == 'pull_request'
     uses: actions/github-script@v7
     with:
       script: |
         const fs = require('fs');
         const comment = `## ⚠️ Breaking API Changes Detected

         The OpenAPI spec has breaking changes that may affect frontend, mobile, and desktop platforms.

         ### Impact
         - Generated TypeScript types have changed
         - Frontend code using old API contracts may break
         - Mobile and desktop platforms are affected

         ### Action Required
         1. Review the type generation logs for specific changes
         2. Update platform code to use new API contracts
         3. Regenerate types locally: \`npm run generate:api-types\`

         See Phase 128 (Backend API Contract Testing) for breaking change details.

         ---
         *Auto-generated by API Type Generation workflow*`;

         github.rest.issues.createComment({
           issue_number: context.issue.number,
           owner: context.repo.owner,
           repo: context.repo.repo,
           body: comment
         });
   ```

2. The breaking change check uses openapi-diff from Phase 128
   - Baseline spec: backend/openapi-baseline.json
   - Fails build on breaking changes (enforces contract stability)

3. PR comments notify developers of required updates

4. Update baseline only on manual approval (via workflow_dispatch)

Do NOT auto-update baseline (would bypass contract stability enforcement)
  </action>
