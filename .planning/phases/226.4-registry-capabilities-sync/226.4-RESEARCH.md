## Summary

Phase 226.4 builds on the solid foundation established in Phase 226.1 (ProviderRegistry models, ProviderRegistryService, ProviderAutoDiscovery) and Phase 226.2 (LUX integration). This phase focuses on three key enhancements:

1. **Capability-Based Routing**: Add `exclude_from_general_routing` flag and capability tags to ModelCatalog for specialized providers (vision-only, audio-only, domain-specific)
2. **Scheduled Auto-Sync**: Integrate APScheduler with ProviderAutoDiscovery for 24-hour polling from LiteLLM/OpenRouter APIs
3. **Provider Health Monitoring**: Track API uptime, error rates, latency and integrate health scores into BPC routing

The existing codebase has excellent infrastructure:
- ✅ ProviderRegistry and ModelCatalog SQLAlchemy models (Phase 226.1)
- ✅ ProviderRegistryService with CRUD operations (415 lines)
- ✅ ProviderAutoDiscovery with sync orchestration (154 lines)
- ✅ DynamicPricingFetcher with LiteLLM/OpenRouter integration
- ✅ Existing APScheduler usage in `backend/core/scheduler.py` (BackgroundScheduler pattern)
- ✅ Existing health monitoring pattern in `backend/core/sync_health_monitor.py`

**Primary recommendation:** Leverage existing patterns (scheduler.py, sync_health_monitor.py), add `exclude_from_general_routing` flag to ModelCatalog, implement APScheduler-based auto-sync with exponential backoff, and create ProviderHealthMonitor service for BPC routing integration.

### Pattern 1: APScheduler AsyncIOScheduler Integration

**What:** Scheduled job execution for auto-sync every 24 hours

**When to use:** Recurring background tasks that need async/await support

**Example:**
```python
# Source: backend/core/scheduler.py (existing pattern)
from apscheduler.schedulers.asyncio import AsyncIOScheduler

class ProviderScheduler:
    """Manages scheduled provider registry sync jobs"""

    def __init__(self):
        self.scheduler = AsyncIOScheduler()
        self.auto_discovery = get_auto_discovery()

    def start(self):
        """Schedule auto-sync every 24 hours"""
        self.scheduler.add_job(
            self._sync_with_error_handling,
            'interval',
            hours=24,
            id='provider_auto_sync',
            max_instances=1,  # Prevent overlapping executions
            coalesce=True,    # Combine missed jobs
        )
        self.scheduler.start()
        logger.info("Provider scheduler started (24-hour polling)")

    async def _sync_with_error_handling(self):
        """Wrapper with error handling and backoff"""
        try:
            result = await self.auto_discovery.sync_providers()
            logger.info(f"Auto-sync completed: {result}")
        except Exception as e:
            logger.error(f"Auto-sync failed: {e}")
            # Health monitor will track failure
```

### Pattern 2: Provider Health Monitoring with Exponential Moving Average

**What:** Track API health (success rate, latency) over sliding window

**When to use:** Real-time health scoring for routing decisions

**Example:**
```python
# Source: backend/core/sync_health_monitor.py (existing pattern)
from collections import deque
from datetime import datetime, timedelta
import logging

class ProviderHealthMonitor:
    """Track provider API health for routing decisions"""

    def __init__(self, window_minutes=5):
        self.window_minutes = window_minutes
        self.call_history = {}  # provider_id -> deque of (timestamp, success, latency_ms)
        self.health_scores = {}  # provider_id -> 0.0-1.0 score

    def record_call(self, provider_id: str, success: bool, latency_ms: float):
        """Record API call outcome for health tracking"""
        timestamp = datetime.now(timezone.utc)

        if provider_id not in self.call_history:
            self.call_history[provider_id] = deque()

        history = self.call_history[provider_id]
        history.append((timestamp, success, latency_ms))

        # Trim old entries outside window
        cutoff = timestamp - timedelta(minutes=self.window_minutes)
        while history and history[0][0] < cutoff:
            history.popleft()

        # Update health score using exponential moving average
        self._update_health_score(provider_id)

    def get_healthy_providers(self, min_score=0.5) -> List[str]:
        """Get providers with health score above threshold"""
        return [p for p, score in self.health_scores.items() if score >= min_score]

    def _update_health_score(self, provider_id: str):
        """Calculate health score from recent history (0.0-1.0)"""
        history = self.call_history.get(provider_id, deque())
        if not history:
            self.health_scores[provider_id] = 1.0  # Default healthy
            return

        # Success rate (70% weight)
        success_count = sum(1 for _, success, _ in history if success)
        success_rate = success_count / len(history)

        # Latency score (30% weight, inverse of latency)
        avg_latency = sum(lat for _, _, lat in history) / len(history)
        latency_score = max(0, 1 - (avg_latency / 5000))  # 5s latency = 0 score

        # Combined score
        self.health_scores[provider_id] = (success_rate * 0.7) + (latency_score * 0.3)
```

### Anti-Patterns to Avoid

- **Scheduling sync too aggressively**: Polling every hour will trigger GitHub rate limits (even for public repos). Use 24-hour interval matching cache duration.
- **Health tracking without cleanup**: Infinite growth of call_history will cause memory leaks. Use deque with sliding window.
- **Hard-coded exclusion lists**: Don't maintain a manual list of "vision-only" models. Use database flag for flexibility.
- **Synchronous job in async scheduler**: Using BackgroundScheduler with async sync_providers() will block. Use AsyncIOScheduler instead.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| **Scheduled job execution** | Custom time.sleep() loop with threading | APScheduler AsyncIOScheduler | Handles cron triggers, job persistence, error recovery, timezone support |
| **Health score calculation** | Custom averaging algorithms | Exponential Moving Average (EMA) | Industry standard for time-series smoothing, built-in weight decay |
| **HTTP async requests** | Custom aiohttp session wrapper | httpx (already used) | Better async/await support, connection pooling, HTTP/2 |
| **Job state persistence** | Custom database table | SQLAlchemyJobStore (APScheduler) | Built-in job persistence, resume after restart |
| **Sliding window** | List with manual cleanup | collections.deque | O(1) append/popleft, thread-safe, no manual index management |

**Key insight:** APScheduler is already in the codebase (scheduler.py). Reuse the pattern instead of building a new scheduler subsystem.

## Common Pitfalls

### Pitfall 1: Polling LiteLLM/OpenRouter Too Aggressively

**What goes wrong:** API rate limits (429 errors), GitHub raw URL throttling, increased bandwidth costs

**Why it happens:** Assuming "more frequent sync = better data", not respecting cache duration

**How to avoid:**
- Set polling interval to 24 hours (matches CACHE_DURATION_HOURS in DynamicPricingFetcher)
- Use `coalesce=True` to combine missed executions
- Set `max_instances=1` to prevent overlapping sync jobs
- Implement exponential backoff after 3 consecutive failures

**Warning signs:** HTTP 429 errors in logs, sync taking >5 minutes, GitHub returning 403

**Code example:**
```python
# BAD: Polls every hour
scheduler.add_job(sync_providers, 'interval', hours=1)

# GOOD: Polls every 24 hours with backoff
scheduler.add_job(
    sync_providers,
    'interval',
    hours=24,
    max_instances=1,
    coalesce=True,
    misfire_grace_time=3600  # 1 hour grace
)
```

### Pitfall 2: Health Tracking Memory Leak

**What goes wrong:** Unbounded growth of call_history causes OOM after days of operation

**Why it happens:** Forgetting to trim old entries outside sliding window

**How to avoid:**
- Use `collections.deque` with maxlen instead of list
- Implement cleanup on every record_call()
- Set reasonable window size (5 minutes is sufficient for health detection)

**Warning signs:** Memory usage growing monotonically, slow health score calculations

**Code example:**
```python
# BAD: Unbounded list
self.call_history = []
self.call_history.append((timestamp, success, latency))

# GOOD: Bounded deque
self.call_history = deque(maxlen=1000)  # Or manual trimming
```

### Pitfall 3: Blocking Async Scheduler with Sync Jobs

**What goes wrong:** Using BackgroundScheduler with async sync_providers() blocks event loop

**Why it happens:** Copying pattern from scheduler.py (which uses sync jobs) for async operations

**How to avoid:**
- Use AsyncIOScheduler for async jobs
- Use BackgroundScheduler for sync jobs
- Always `await` coroutines in job functions

**Warning signs:** "This event loop is already running" errors, jobs not executing

**Code example:**
```python
# BAD: BackgroundScheduler with async job
from apscheduler.schedulers.background import BackgroundScheduler
scheduler = BackgroundScheduler()
scheduler.add_job(sync_providers, 'interval', hours=24)  # sync_providers is async!

# GOOD: AsyncIOScheduler with async job
from apscheduler.schedulers.asyncio import AsyncIOScheduler
scheduler = AsyncIOScheduler()
scheduler.add_job(sync_providers, 'interval', hours=24)
```

### Pitfall 4: Database Query in Hot Path (BPC Routing)

**What goes wrong:** Querying ModelCatalog for every routing request degrades performance

**Why it happens:** Checking `exclude_from_general_routing` flag without caching

**How to avoid:**
- Cache excluded models in memory (refresh on sync)
- Use ProviderRegistryService.get_provider_stats() for batch queries
- Add Redis cache if scalability needed (beyond Phase 226.4 scope)

**Warning signs:** BPC routing latency >100ms, slow queries in DB logs

**Code example:**
```python
# BAD: Query database on every routing request
def get_ranked_providers(self, complexity, **kwargs):
    for model_id in pricing_cache:
        model = db.query(ModelCatalog).filter_by(model_id=model_id).first()  # SLOW
        if model.exclude_from_general_routing:
            continue

# GOOD: Cache excluded models in memory
class BYOKHandler:
    def __init__(self):
        self.excluded_models = set()
        self._refresh_excluded_cache()

    def _refresh_excluded_cache(self):
        # Called on sync and startup
        with get_db_session() as db:
            excluded = db.query(ModelCatalog.model_id).filter(
                ModelCatalog.exclude_from_general_routing == True
            ).all()
            self.excluded_models = {m[0] for m in excluded}
```

### Pitfall 5: Hard-Coded Capability Tags

**What goes wrong:** Maintaining manual list of "vision", "audio", "code" models breaks on new releases

**Why it happens:** Not using database metadata for capabilities

**How to avoid:**
- Store capability tags in ModelCatalog.model_metadata JSON field
- Auto-detect from provider API responses (LiteLLM has mode field)
- Allow manual override via admin API

**Warning signs:** Frequent code changes for new models, outdated exclusion lists

**Code example:**
```python
# BAD: Hard-coded list
VISION_ONLY_MODELS = ["claude-3-haiku-vision", "gpt-4-vision-preview"]
if model_id in VISION_ONLY_MODELS:
    continue

# GOOD: Database metadata
model = db.query(ModelCatalog).filter_by(model_id=model_id).first()
if model and model.model_metadata.get("capabilities") == ["vision"]:
    continue
```

class SyncHealthMonitor:
    def __init__(self):
        self.expected_sync_interval_minutes = 30
        self.error_window_minutes = 5

    def check_health(self, db: Session) -> Dict[str, Any]:
        health_status = {
            "status": "healthy",
            "last_sync": None,
            "sync_age_minutes": None,
            "recent_errors": 0,
            "checks": {}
        }

        # Check 1: Last sync age
        sync_check = self._check_last_sync(db)
        health_status["checks"]["last_sync"] = sync_check

        # Determine overall status
        failed_checks = [k for k, v in health_status["checks"].items() if not v.get("healthy")]
        degraded_checks = [k for k, v in health_status["checks"].items() if v.get("degraded")]

        if failed_checks:
            health_status["status"] = "unhealthy"
        elif degraded_checks:
            health_status["status"] = "degraded"

        return health_status

### Existing Auto-Discovery Pattern (provider_auto_discovery.py)

```python
# Source: backend/core/provider_auto_discovery.py (lines 17-55)
async def sync_providers(self) -> Dict[str, int]:
    """Fetch latest pricing and sync to registry"""
    logger.info("Starting provider sync from DynamicPricingFetcher")

    # 1. Refresh pricing cache
    await self.pricing_fetcher.refresh_pricing(force=True)
    logger.info(f"Pricing cache refreshed: {len(self.pricing_fetcher.pricing_cache)} models")

    # 2. Extract unique providers from pricing_cache
    providers_synced = 0
    models_synced = 0
    seen_providers = set()

    for model_id, pricing in self.pricing_fetcher.pricing_cache.items():
        # Extract and upsert provider
        provider = self._extract_provider_from_model(model_id, pricing)
        if provider and provider['provider_id'] not in seen_providers:
            self.registry.upsert_provider(provider)
            seen_providers.add(provider['provider_id'])
            providers_synced += 1

        # Extract and upsert model
        model = self._extract_model_from_pricing(model_id, pricing)
        if model:
            self.registry.upsert_model(model)
            models_synced += 1

    return {"providers_synced": providers_synced, "models_synced": models_synced}
```

## Open Questions

### Question 1: Should scheduler start automatically on application boot?

**What we know:** AgentScheduler (scheduler.py) starts in `__init__`, no explicit start call needed

**What's unclear:** Should ProviderScheduler follow same pattern or require manual start?

**Recommendation:** Auto-start on application boot (same as AgentScheduler), but add environment variable toggle:
```python
PROVIDER_AUTO_SYNC_ENABLED = os.getenv("PROVIDER_AUTO_SYNC_ENABLED", "true").lower() == "true"

if PROVIDER_AUTO_SYNC_ENABLED:
    scheduler = ProviderScheduler()
    scheduler.start()
```

### Question 2: Health check endpoint integration

**What we know:** Existing `/health/ready` endpoint checks DB and disk health

**What's unclear:** Should provider health be part of readiness probe?

**Recommendation:** No. Provider health is **not** a readiness concern (app can function without providers). Add separate `/health/providers` endpoint for monitoring:
- **Readiness** (`/health/ready`): DB connection, disk space
- **Provider health** (`/health/providers`): API availability, last sync time
- **Liveness** (`/health/live`): Application responding (always 200)

### Question 3: Health score persistence

**What we know:** Health scores decay over time (EMA), need to survive restarts

**What's unclear:** Should health scores be stored in database or in-memory only?

**Recommendation:** In-memory only for this phase. Health scores are short-lived (5-minute window), rebuilding after restart is acceptable. Future phases can add Redis persistence if needed.

### Question 4: Exclusion flag auto-detection

**What we know:** LiteLLM provides `mode` field ("chat", "completion", "vision")

**What's unclear:** Should we auto-set `exclude_from_general_routing` based on mode?

**Recommendation:** Auto-detect for obvious cases, allow manual override:
```python
# Auto-exclude vision-only models
if pricing.get("mode") == "vision":
    model_data["exclude_from_general_routing"] = True
    model_data["model_metadata"]["capabilities"] = ["vision"]

# But allow manual override via admin API
```
