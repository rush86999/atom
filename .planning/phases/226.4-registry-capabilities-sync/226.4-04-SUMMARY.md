## Decisions Made

- **Capability-specific quality scores:** When `required_capability` is set, use `get_capability_score(model_id, required_capability)` for specialized quality assessment. When not set, fall back to `get_quality_score(model_id)` for backward compatibility.

- **Excluded models logic:** Models with `exclude_from_general_routing=True` are ONLY excluded when `required_capability` is None (general routing). This allows specialized routing to computer_use, vision, or tools when explicitly requested.

- **Health threshold:** Set to 0.5 on a 0.0-1.0 scale. This represents a balanced threshold where providers with <50% overall health (70% success rate + 30% latency score) are excluded from routing.

- **Optimistic default for unknown providers:** Providers with no health history return `1.0` (healthy) to allow new providers to be tried. This prevents cold-start problem where new providers would never be selected.

- **Health recording error handling:** All `health_monitor.record_call()` calls wrapped in try/except to prevent monitoring errors from affecting primary LLM request flow. This ensures reliability even if health monitoring fails.

- **Request timing tracking:** Added `time.time()` calls before API calls to measure latency in milliseconds. Used for health scoring (latency component) and performance monitoring.

## Implementation Highlights

### Capability-Based Routing Flow

1. **Request received** with `required_capability` parameter (e.g., "computer_use")
2. **Capability filter** - `get_ranked_providers()` checks ModelCatalog.capabilities JSON field
3. **Excluded models filter** - Models with `exclude_from_general_routing=True` skipped (only when no required_capability)
4. **Health filter** - Providers with `health_score < 0.5` excluded
5. **Quality scoring** - `get_capability_score(model_id, required_capability)` used for specialized quality
6. **BPC ranking** - Models ranked by Benchmark-Price-Capability score
7. **Provider selection** - Top-ranked provider/model selected

### Health Monitoring Flow

1. **Request started** - `request_start = time.time()`
2. **API call executed** - LLM provider called
3. **Success/Failure detected** - Try/except blocks
4. **Latency calculated** - `(time.time() - request_start) * 1000` ms
5. **Health recorded** - `health_monitor.record_call(provider_id, success=bool, latency_ms=float)`
6. **Health score updated** - ProviderHealthMonitor recalculates EMA-based score
