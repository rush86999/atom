## User Decisions (LOCKED)

These decisions are LOCKED and must be honored by the plan:

### Decision 1: Capability-Based Routing System
**Decision:** BYOK/BPC system should handle specialized models through capability flags, not just exclusion.

**Rationale:**
- Simple exclusion is insufficient - we need positive capability matching
- Models can have multiple capabilities (chat + tools, vision + tools, etc.)
- Routing should match task requirements to model capabilities
- Enables more sophisticated routing decisions

**Implementation Requirements:**
- Add `capabilities` column to ModelCatalog (JSON array: ["chat"], ["vision"], ["computer_use", "browser_use"], etc.)
- Detect capabilities from model metadata (mode, provider, model name patterns)
- BYOKHandler must filter models by required capabilities BEFORE applying BPC scoring
- Models lacking required capabilities are excluded from that routing decision

---

### Decision 2: LUX is Specialized (Computer Use + Browser Use Only)
**Decision:** LUX model (lux-1.0) is specialized for computer use and browser use ONLY.

**Rationale:**
- LUX is based on Claude 3.5 Sonnet but fine-tuned for computer control
- Not optimized for regular chat, summarization, or general text tasks
- Should only appear in routing for computer_use/browser_use tasks
- Should be EXCLUDED from general text routing (chat, summarization, etc.)

**Implementation Requirements:**
- LUX capabilities: `["computer_use", "browser_use"]` (NO "chat" capability)
- Auto-detection: `model_name.startswith("lux")` → computer_use capability
- `exclude_from_general_routing` = true (derived from lack of "chat" capability)
- BYOKHandler.get_ranked_providers(task_type="computer_use") includes LUX
- BYOKHandler.get_ranked_providers(requires_vision=False, task_type=None) excludes LUX

---

### Decision 3: Capability-Aware Quality Scores
**Decision:** Benchmarks should support capability-specific quality scores, not just generic model scores.

**Rationale:**
- A model excellent at chat might be mediocre at vision
- A model specialized for computer use should have high computer_use score
- BPC routing should use capability-specific scores for ranking
- Enables more accurate routing decisions based on actual capability performance

**Implementation Requirements:**
- Add `MODEL_CAPABILITY_SCORES` dictionary to benchmarks.py
- Structure: `{"capability": {"model_id": score}}` (e.g., `{"computer_use": {"lux-1.0": 95}}`)
- Add `get_capability_score(model_id, capability)` function
- BYOKHandler.get_ranked_providers() uses `get_capability_score(model_id, required_capability)`
- Fallback to generic MODEL_QUALITY_SCORES if capability-specific score missing

**Examples:**
- Vision tasks: GPT-4o (95) > Claude 3.5 Sonnet (90) > LUX (85)
- Computer use: LUX (95) > Claude 3.5 Sonnet (85)
- Tools: Claude 3.5 Sonnet (93) > GPT-4o (95)
- Chat: GPT-4o (95) = Claude 3.5 Sonnet (92)

---

### Decision 4: Auto-Sync and Health Monitoring
**Decision:** Provider registry should auto-sync from provider APIs and track health for routing decisions.

**Rationale:**
- Provider models change frequently (new models, deprecated models)
- Health status impacts routing (unhealthy providers should be deprioritized)
- Automated sync reduces manual maintenance

**Implementation Requirements:**
- ProviderScheduler using AsyncIOScheduler (24-hour polling interval)
- ProviderHealthMonitor with sliding window (5-minute) and EMA scoring
- BYOKHandler filters providers by health score (min 0.5)
- Health status API endpoints for monitoring
