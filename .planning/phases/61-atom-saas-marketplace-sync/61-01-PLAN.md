# Objective
Build a background sync service that periodically pulls skills and categories from Atom SaaS and populates the local cache (SkillCache, CategoryCache models). This provides near real-time marketplace data without user intervention.

# Key decisions
- **APScheduler**: Lightweight, in-process scheduler (no Redis/Celery needed)
- **15-minute interval**: Balance between freshness and API load
- **Batch size 100**: Optimal for API performance vs memory
- **Exponential backoff**: 1s, 2s, 4s, 8s, 16s max on failures
- **TTL invalidation**: 5-minute cache expiry ensures fresh data
- **Sync state tracking**: Single-row table for last sync metadata

# Edge cases to handle
- Empty marketplace (0 skills): Sync should complete gracefully
- Network timeout: Retry with exponential backoff
- API rate limit (429): Respect Retry-After header or backoff 60s
- Duplicate skills: Upsert using skill_id as unique key
- Concurrent sync requests: Return 503 if sync already in progress
- Database connection loss: Retry DB operations, log errors
- Scheduler shutdown during sync: Allow current sync to complete, don't start new ones

# Performance considerations
- Batch fetching reduces API calls (100 skills per request)
- Cache invalidation uses bulk delete (WHERE expires_at < now)
- SyncState updates use single-row table (no query bloat)
- APScheduler runs in background thread (doesn't block API requests)

# Security considerations
- Admin endpoints require AUTONOMOUS maturity
- API token never logged (use asterisks in logs)
- Audit trail for all manual sync triggers
- Input validation on sync trigger parameters
