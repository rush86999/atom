# Key decisions
- **Default strategy: remote_wins** - Atom SaaS is source of truth
- **Automatic merge fields**: description, tags, examples, metadata (safe to update)
- **Manual resolution fields**: code, dependencies, command (critical changes)
- **Conflict detection**: Compare skill_id + version + content_hash
- **Conflict types**: VERSION_MISMATCH, CONTENT_MISMATCH, DEPENDENCY_CONFLICT
- **Resolution tracking**: ConflictLog stores original, resolution, timestamp

## Task 2: Implement conflict detection logic
**Objective**: Detect when skills conflict between local and remote

**Implementation**:
- Create ConflictResolutionService in `backend/core/conflict_resolution_service.py`
- Methods:
  - `__init__(Session)`: Initialize with DB session
  - `detect_skill_conflict(local_skill, remote_skill)`: Compare skills and return conflict type
  - `compare_versions(local, remote)`: Check if versions match
  - `compare_content(local, remote)`: Check if code/content matches
  - `compare_dependencies(local, remote)`: Check if dependencies match
- Conflict detection logic:
  - VERSION_MISMATCH: Different version strings
  - CONTENT_MISMATCH: Different content_hash or code
  - DEPENDENCY_CONFLICT: Different Python/npm packages
  - OTHER: Catch-all for other differences
- Add severity scoring:
  - LOW: Description, tags, examples changed
  - MEDIUM: Metadata, parameters changed
  - HIGH: Dependencies, version changed
  - CRITICAL: Code, command, security-sensitive fields changed

**Acceptance criteria**:
- Conflict detection identifies all conflict types
- Severity scoring is accurate
- Comparison methods handle null values
- Detection logged for debugging

## Task 3: Implement merge strategies (remote_wins, local_wins, merge, manual)
**Objective**: Create 4 strategies for resolving conflicts

**Implementation**:
- Add strategy methods to ConflictResolutionService:
  - `remote_wins(local, remote)`: Return remote skill data
  - `local_wins(local, remote)`: Return local skill data
  - `merge(local, remote)`: Merge fields intelligently

- Add `resolve_conflict(conflict_id, strategy, resolved_by)` method
- Add `auto_resolve_conflict(local_skill, remote_skill, strategy)` method

**Acceptance criteria**:

# Edge cases to handle
- Skill exists locally but not remotely: Keep local, add to ConflictLog as ORPHANED_LOCAL
- Skill exists remotely but not locally: Normal sync (no conflict)
- Multiple conflicts for same skill: Resolve in timestamp order, merge all
- Circular merge (local → remote → local): Detect loop, stop after 3 iterations
- Merge strategy creates invalid skill: Validate merged skill, reject if invalid
- Manual conflict never resolved: Send alert after 7 days, escalate after 30 days
- Concurrent resolution attempts: Lock conflict row during resolution

# Performance considerations
- Conflict detection doesn't block sync (process in background)
- Bulk resolution uses transaction (all or nothing)
- ConflictLog query uses indexes (skill_id, severity, resolved_at)
- Diff summary generated on-demand (not stored)

# Security considerations
- Admin endpoints require AUTONOMOUS maturity
- Input validation on strategy parameter
- Audit trail for all manual resolutions
- Rate limiting on bulk resolution (max 100 at a time)
