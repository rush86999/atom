Implement WebSocket connection to Atom SaaS for real-time skill, category, and rating updates. WebSocket provides near-instant updates compared to 15-minute polling, with automatic fallback to polling on disconnect.

# Key decisions
- **websockets library**: Async, native Python WebSocket implementation
- **Heartbeat 30 seconds**: Detect stale connections quickly
- **Exponential backoff**: Reconnect with increasing delays (max 16s)
- **Message format**: JSON with `type` field and `data` payload
- **Fallback to polling**: If WebSocket fails 3 times consecutively, switch to polling for 1 hour
- **Single connection**: One WebSocket connection for all updates (not one per client)

## Task 2: Implement heartbeat and reconnection logic
**Objective**: Keep connection alive and auto-reconnect on disconnect

**Implementation**:
- Add heartbeat method to AtomSaaSWebSocketClient
- Send `{"type": "ping"}` every 30 seconds
- Expect `{"type": "pong"}` response within 10 seconds
- If no pong received, disconnect and attempt reconnection
- Implement exponential backoff reconnection:
  - Attempt 1: 1 second delay
  - Attempt 2: 2 seconds delay
  - Attempt 3: 4 seconds delay
  - Attempt 4: 8 seconds delay
  - Attempt 5+: 16 seconds delay (max)
- Update WebSocketState with reconnect attempts and disconnect reason
- Stop reconnection after 10 consecutive failures (require manual intervention)

**Acceptance criteria**:
- Heartbeat sent every 30 seconds
- Connection closed if no pong received
- Reconnection uses exponential backoff
- WebSocketState updated with reconnect attempts
- Reconnection stops after 10 failures

# Edge cases to handle
- WebSocket server unavailable: Fall back to polling immediately
- Connection dropped during message: Reconnect and request missed messages (if API supports)
- Malformed message: Log warning, skip message, continue processing
- Message flood (1000+ msg/sec): Rate limit to 100 msg/sec, log warning
- Stale connection (no messages for 2 minutes): Disconnect and reconnect
- Concurrent reconnect requests: Only one reconnection attempt at a time
- Server sends unknown message type: Log warning, skip message

# Performance considerations
- Single WebSocket connection (not one per client)
- Async message handling doesn't block sync operations
- Rate limiting protects against message floods
- Heartbeat lightweight (small JSON ping/pong)
- Reconnection backoff reduces server load

# Security considerations
- Admin endpoints require AUTONOMOUS maturity
- Message validation prevents injection attacks
- Rate limiting prevents DoS
- WebSocket URL uses wss:// (TLS) in production
- Token validation on connection (Bearer token in query param or header)
