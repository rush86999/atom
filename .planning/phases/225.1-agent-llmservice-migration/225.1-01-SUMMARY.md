## Key Decisions

### Decision 1: Only One Method Needs to Be Added to LLMService
**Context:** Gap analysis revealed `analyze_query_complexity()` is not directly exposed as top-level LLMService method
**Decision:** Add `analyze_query_complexity()` as top-level LLMService method with simple delegation to `self.handler.analyze_query_complexity()`
**Rationale:**
- Cleaner API than accessing via `llm_service.handler.analyze_query_complexity()`
- Maintains consistency with other exposed methods
- Zero breaking changes (additive only)
- 5-minute implementation (simple delegation)

### Decision 2: Migration Order from Simplest to Most Complex
**Context:** 4 services need migration, ranging from simple dependency injection to complex ReAct loop
**Decision:** Migrate in order of complexity:
1. CanvasSummaryService (LOW) - Single dependency change
2. EpisodeSegmentationService (LOW) - Single dependency change
3. AgentExecutionService (LOW) - Replace instantiation, 2 method calls
4. GenericAgent (MEDIUM) - 3 method calls, instructor integration

**Rationale:**
- Build confidence with easy wins first
- Catch integration issues early on simpler services
- GenericAgent has most tests (53), so save for last when migration pattern is proven

### Decision 3: No Architectural Changes Needed
**Context:** User requested "LLMService is the single source of truth" for easier management
**Decision:** LLMService remains thin wrapper around BYOKHandler (no architectural changes)
**Rationale:**
- LLMService already delegates to BYOKHandler internally
- Zero performance overhead (wrapper pattern)
- BYOKHandler remains internal implementation detail
- Cleaner API for external consumers
