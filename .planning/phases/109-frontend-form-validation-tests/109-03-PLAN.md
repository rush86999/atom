Create `frontend-nextjs/components/canvas/__tests__/form-error-messages.test.tsx` testing error message behavior:

**CRITICAL Pattern from Research (Pitfall 2):**
InteractiveForm only clears errors on next submit, NOT on input change. Tests must reflect this.

**Test Structure:**
```typescript
import { render, screen, waitFor } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import '@testing-library/jest-dom';
import { InteractiveForm } from '../InteractiveForm';

const mockCanvasAPI = {
  getState: jest.fn(),
  getAllStates: jest.fn(),
  subscribe: jest.fn(),
  subscribeAll: jest.fn()
};

beforeEach(() => {
  (window as any).atom = { canvas: mockCanvasAPI };
  jest.clearAllMocks();
});

afterEach(() => {
  delete (window as any).atom;
});
```

**Test Groups (400+ lines):**

1. **Error Message Location and Visibility** (8 tests, 100 lines)
   - Error appears below the field (test for text-red-500 class)
   - Error includes AlertCircle icon
   - Multiple errors show for multiple fields
   - Form-level error shows for submission failures
   - Error is visible immediately after submit
   - Error persists when user starts typing
   - Error disappears on next submit after correction
   - No duplicate error messages

2. **Required Field Error Messages** (6 tests, 80 lines)
   - "{label} is required" format
   - Label matches field label exactly
   - Error for empty required field
   - Error for whitespace-only required field
   - Custom error message via validation.custom
   - No error for optional empty field

3. **Format Validation Error Messages** (8 tests, 120 lines)
   - Email format error: "{label} format is invalid" or custom message
   - Pattern validation uses custom message when provided
   - Pattern validation uses default when custom omitted
   - Phone format error message
   - URL format error message
   - Custom regex error shows pattern-specific message
   - Special characters in custom messages display correctly

4. **Range Validation Error Messages** (8 tests, 120 lines)
   - Min error: "{label} must be at least {min}"
   - Max error: "{label} must be at most {max}"
   - Range error when both min and max
   - Number type vs text type handling
   - Exact boundary values show no error
   - Min-1 and max+1 show errors

5. **Error Clearing Behavior** (6 tests, 80 lines)
   - **CRITICAL:** Error persists after user types input
   - Error clears only on next submit after correction
   - All errors clear when form becomes valid
   - Individual error clears for corrected field
   - Form-level error clears on next submit
   - No errors when form is valid from start

**Key Anti-Patterns to Avoid:**
```typescript
// ❌ WRONG: Expecting error to clear on type/input change
await user.type(input, 'valid input');
expect(screen.queryByText(/error/i)).not.toBeInTheDocument();

// ✅ CORRECT: Resubmit to clear error
await user.type(input, 'valid input');
await user.click(submitButton);
await waitFor(() => {
  expect(screen.queryByText(/error/i)).not.toBeInTheDocument();
});
```

**Test Groups (350+ lines):**

   - Button remains disabled until submission completes

2. **Success State Feedback** (6 tests, 90 lines)
   - Success message appears after successful submit
   - Success message shows green checkmark icon
   - Success message reads "Submitted successfully!"
   - Form is replaced by success view
   - Success message auto-hides after 3 seconds
   - Form reappears after success message hides

3. **Error State Feedback** (8 tests, 120 lines)
   - Form-level error appears on submission failure
   - Form-level error uses error styling (red background)
   - Form-level error message: "Submission failed. Please try again."
   - Form remains visible after error
   - User can retry submission after error
   - Field errors and form errors can coexist
   - Error state doesn't block form editing
   - Network error message is user-friendly

4. **Accessibility Feedback** (6 tests, 80 lines)

5. **Interactive Feedback Scenarios** (4 tests, 70 lines)
